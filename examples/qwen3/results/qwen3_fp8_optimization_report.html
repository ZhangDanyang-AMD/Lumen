3. 已实施的优化点

已完成 ① frozen-weight FP8 缓存（核心，−14%）

LoRA 训练里 base weight 冻结，其 FP8 量化结果永远不变，但原实现每次 forward（以及 gradient-checkpointing 的重算 forward）都重新量化一遍 —— 505 ≈ 2×253 次/step，约 43.5 GB/step 的 contiguous 拷贝。

  • 做法：首次 forward 时（FSDP 已 gather 出完整权重）量化一次，缓存 _fp8_weight_data/_fp8_weight_scale 到 module；之后所有 forward（含重算、跨 step）复用，跳过 _quant_blockwise2d_weight。
  • 同时缓存转置权重：blockwise2d 的 DGrad 每步做 weight_data.t().contiguous()（~7.6 GB/step）；把冻结权重的转置 FP8+scale 也缓存（挂在稳定缓存张量上，forward 经 ctx 传给 backward，零 autograd 签名改动）。
  • 关键坑：FSDP 下被 gather 的冻结权重 view 会误报 requires_grad=True（flat-param 传染），所以 frozen 判定放在 patch 时（FSDP 包装前，此时 PEFT 已冻结 base，requires_grad 可靠）。
  • 开关：cache_frozen_weight（QuantConfig/LumenConfig，默认关）/ --cache-frozen-weight / launcher CACHE_FROZEN_WEIGHT=1。
  • 代价：缓存常驻完整 FP8 权重，8B ≈ +8 GB/GPU（192GB 上没问题）；70B FSDP 会 OOM，不要开。

已完成 ①b 跳过冻结权重 WGrad（最大单项，−17%）

LoRA 下 base weight 冻结，但 FSDP 把它的 requires_grad 误标成 True，autograd 仍完整计算它的 WGrad（dequant→requant + 转置 + Triton blockscale GEMM），结果却被丢弃 —— 纯浪费。由 cache_frozen_weight 统一控制：缓存张量带 skip 标记，forward 经 ctx 传给 backward，直接 grad_weight=None，整条 WGrad 不执行。

  • step time 2390 → 1912 ms（再降 ~17pt，累计 −31%），copy_ 0.98 → 0.34 s。
  • DGrad（传给输入的梯度）与 LoRA 梯度完全不受影响 → loss 逐位一致（已加测试验证）。
  • 这才是 WGrad 的正确处理：冻结本就不该算梯度，不是去"加速"它而是"不做它"。

已完成 ② GEMM tuned config（gfx942，~3%）

镜像里的 a8w8_blockscale_tuned_gemm.csv 零 gfx942 条目（全是 gfx950），MI308X 上 blockscale GEMM 全走默认启发式 kernel。为 Qwen3 的 5 个 transformer 层 shape（M=2048）在 gfx942/80CU 上 tune，存入 examples/qwen3/configs/a8w8_blockscale_tuned_gemm.csv。

  • 启用：USE_TUNED_GEMM=1（launcher，默认关）→ 把 AITER_CONFIG_GEMM_A8W8_BLOCKSCALE 指向仓库内 csv。
  • 收益小：tuner 给 5 个 shape 都选了 kernelId 0 ≈ 默认已用的 kernel，提速仅 ~3%，step time 几乎不变。结论：GEMM kernel 选型不是瓶颈。
  • 注意：只匹配 gfx942/cu_num=80/M=2048；在 gfx950 上启用会丢掉镜像 gfx950 配置，非 MI308X 不要开。

已完成 ③ Profiler + copy 追踪工具

给 qwen3 训练脚本加了 env 门控的 torch.profiler（LUMEN_PROF_START/END/OUTPUT/TRACE）和 copy 归因（LUMEN_COPY_TRACE=1，monkey-patch copy_/contiguous 按 lumen 调用栈聚合）。正是它定位到真正的 copy 热点（权重重量化），纠正了最初对 WGrad 的误判。

已回退 ④ WGrad requant 加速（被 ①b 取代）

最初尝试加速 blockwise2d WGrad 的 dequant→requant 往返，SNR 通过但零收益 —— copy trace 证实 LoRA 下 base WGrad 被丢弃。正确做法不是加速而是整条跳过（见 ①b）。教训：先用 copy trace 定位再动手。
