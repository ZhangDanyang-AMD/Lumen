② 每一步做了什么、为什么

#优化一句话原理step类型
1缓存冻结权重的 FP8 量化LoRA base 冻结，量化结果不变，却每次 forward 重算 → 只算一次并复用2645无损
2缓存 DGrad 的转置权重反向 DGrad 每步对冻结权重做转置拷贝 → 转置结果也缓存2390无损
3跳过冻结权重的 WGrad冻结权重的梯度本就丢弃，FSDP 却仍整条算 → 直接不算1912无损
4关 gradient checkpointing反向不再重算 forward（GEMM 758→506/step），消除重算气泡1610换显存
5SHARD_GRAD_OP + overlap参数常驻，反向不再 all-gather（73→37 次/step），消除通信气泡1542换显存
6bpreshuffle GEMM预重排冻结权重布局，blockscale GEMM 提速 ~2.6x（FP8 GEMM 1028→396ms），bit-identical960无损

③ 瓶颈构成的变化

优化前 GPU 时间被拷贝/重算/通信占去近一半；优化后这些基本消失，剩下的几乎都是不可省的 GEMM 计算。

优化前（FP8 baseline）
GEMM 37% copy 23% elem 15% 通信 13% attn
优化后（最优配置）
GEMM ~60% copy 通信 attn
FP8 GEMM 拷贝 copy_ elementwise 量化 FSDP 通信 attention

⑤ 试过但没采用

尝试结果
GEMM tuned config (gfx942)仅 ~3%，默认 kernel 已近最优；可选开关 USE_TUNED_GEMM
加速 WGrad 的 requant零收益（冻结权重 WGrad 本就被丢弃）→ 改为直接跳过（步骤 3）
SHARD_GRAD_OP（grad-ckpt 开着时）反而更慢；只有关掉 grad-ckpt 才有效（步骤 5）
NO_SHARD已废弃，本配置报错，不可用
lm_head 大 GEMM tunetune 成本极高（>40min 未完成），且仅占 ~1%，放弃

⑥ 怎么用

全部优化（8B，显存充足）：
CACHE_FROZEN_WEIGHT=1 BPRESHUFFLE=1 GRAD_CKPT=0 SHARDING=shard_grad_op \
LIMIT_ALL_GATHERS=0 FORWARD_PREFETCH=1 MODE=fp8_blockwise2d \
  bash examples/qwen3/run_qwen3_fsdp_mi308.sh
显存吃紧 / 大模型：去掉 GRAD_CKPT=0 和 SHARDING，仍保留库层无损优化（约 −31%）。
