Lumen: AMD-Native
Quantized Training for LLMs

A lightweight engine that manages the quantized training lifecycle — forward, backward, optimizer, and communication — with one-line non-invasive patching on AMD Instinct hardware. Now with VERL FSDP2 & Megatron RL, SGLang and vLLM rollout on ROCm.

Key Features

Non-invasive FP8 quantized training with high-performance AMD-native kernels

⚡

Non-invasive Patching

quant.enable(model) patches any model for FP8 training without changing model code or checkpoint format. One line, zero model modifications.

⚙

Unified QuantConfig

A single configuration object governs format (E4M3, E5M2, MXFP8, Hybrid), scaling strategy (Dynamic, Delayed, Blockwise), and all precision knobs.

▲

Up to 2x Peak FLOPS

FP8 on MI300X achieves ~2,615 TFLOPS vs ~1,307 TFLOPS for BF16. End-to-end training speedup of 1.4-1.7x depending on workload.

♺

AITER Kernels

High-performance fused operators with ASM -> CK -> Triton fallback chain. Covers attention, GEMM, normalization, RoPE, MoE, MLP, cross-entropy.

☷

MORI Communication

Native RDMA + GPU communication with FP8 all-gather (~2x bandwidth saving) and compute-communication overlap (AG/fwd, RS/bwd).

⚗

Backend-Agnostic

Same quantization semantics across FSDP, FSDP2, and Megatron-LM backends. Supports SFT, pre-training, and RL workloads (GRPO, PPO, DAPO) with SGLang or vLLM rollout.

Architecture

Lumen owns the quantized training lifecycle and delegates everything else to the training backend

Quick Start

Enable quantized training with a single line of code

Install
# User install
pip install lumen[all]

# Developer install
git clone git@github.com:ZhangDanyang-AMD/Lumen.git
cd Lumen
pip install -e ".[dev]"
Non-invasive Patching
import lumen.quantize as quant
from lumen.quantize import QuantConfig, QuantFormat, ScalingType

# One line — patches model for FP8 training
quant.enable(model, format="fp8_e4m3", scaling="delayed")

# Training loop unchanged
output = model(input_ids)
loss.backward()
optimizer.step()
Full Config
config = QuantConfig(
    format=QuantFormat.FP8_E4M3,
    scaling=ScalingType.DELAYED,
    amax_algo=AmaxAlgo.MAX,
    history_len=16,
    quantize_activation=True,
    quantize_grad="fp8",
)
quant.enable(model, config=config)
RL Training (VERL + FSDP2 + SGLang/vLLM)
# VERL + FSDP2 + SGLang with FP8 Param Manager
# -25% peak VRAM vs BF16, 1.3x faster vs Megatron
FP8_PARAM_MANAGER=1 \
NUM_GPUS=4 ROLLOUT_TP=2 \
bash examples/rl/verl/run_grpo_fsdp2.sh

# Also works with vLLM rollout on ROCm
bash examples/rl/verl/run_grpo_fsdp2_vllm.sh

Key Findings

1. FP8PM saves memory with both offload modes (after clone fix)

_FP8LinearFunc.forward previously pinned FSDP2's allgathered weight tensors on GPU, causing +44% memory regression with offloading (48→69 GB). Fix: ctx.save_for_backward(fp8_weight.clone(), scale) in lumen/quantize/fp8_params.py.

ModeBF16FP8PM (after fix)Savings
With offload48.06 GB45.50 GB-5%
Without offload73.49 GB54.87 GB-25%

2. FSDP2 is more memory-efficient than Megatron for BF16

With offloading, FSDP2 achieves 32% lower peak memory (48 vs 71 GB) and 13% higher throughput (797 vs 704 tok/s) compared to Megatron for the same model.

3. Megatron FP8PM now works (on-the-fly quantization)

FP8ParamManager was extended to target Megatron's ColumnParallelLinear/RowParallelLinear. Unlike FSDP2 (in-place FP8 quantization), Megatron uses on-the-fly quantization: parameters stay BF16 for optimizer/DDP compatibility, while _FP8MegatronLinearFunc quantizes to FP8 during forward and saves compact FP8 data in the autograd graph. In-place quantization crashed Megatron's distributed optimizer. Injection gated by FP8_PARAM_MANAGER=1.

LoRA with Megatron remains unsupported — PEFT expects HuggingFace model structure, not Megatron parallel layers.

4. vLLM V1 works on ROCm after get_device_uuid fix

vLLM V1 was previously broken on ROCm — both FSDP2+vLLM and Megatron+vLLM hung indefinitely during weight updates. The root cause was a missing @with_amdsmi_context decorator on RocmPlatform.get_device_uuid() in vllm/platforms/rocm.py. Without it, amdsmi was uninitialized in VERL worker processes, producing a fallback UUID that didn't match the EngineCore's real hardware UUID, causing mismatched ZMQ IPC socket paths and a deadlock. Fix: Add @with_amdsmi_context to get_device_uuid() — a one-line change.

Note: Rollout TP≥2 may still hang during initialization on ROCm due to a separate spawn multiprocessing issue. Use TP=1 until upstream vLLM addresses this.

5. FSDP2 outperforms Megatron across both rollout engines

FSDP2 consistently achieves 1.3× higher throughput than Megatron for the same model and comparable batch size.

6. vLLM FP8PM: savings visible at low gpu_memory_utilization

With gpu_util=0.4, vLLM reserves ~101 GB, masking FP8PM savings. Throughput improves +6% from reduced memory bandwidth. With gpu_util=0.1:

  • Peak VRAM: 29.80 GB (FP8PM) vs 34.26 GB (BF16) — 13% less
  • FSDP memory after init: 0.12 GB (FP8PM) vs 0.46 GB (BF16) — 74% less
  • Throughput: 477 tok/s (FP8PM) vs 445 tok/s (BF16) — 7% faster

7. USE_8BIT_ADAM env var does not change the optimizer

The USE_8BIT_ADAM=1 flag in LumenConfig is a hint — VERL's build_optimizer uses FSDPOptimizerConfig (default: torch.optim.AdamW). To use 8-bit Adam, set actor_rollout_ref.actor.optim.optimizer=AdamW8bit in VERL config. Note: bitsandbytes AdamW8bit is incompatible with FSDP2 DTensor — use torchao.optim if DTensor-compatible 8-bit Adam is needed.

8. Megatron FP8PM: memory savings visible, throughput regression with vLLM

With SGLang: Megatron FP8PM shows 50.06 GB vs 70.52 GB (BF16) — 29% less peak VRAM. Throughput drops from 704 to 369 tok/s due to on-the-fly quantization overhead.

With vLLM (gpu_util=0.3): 86.89 GB vs BF16's 85.01 GB — negligible difference since vLLM KV cache dominates. Throughput is 291 tok/s vs BF16's 338 tok/s (-14% regression). For throughput-sensitive workloads, FSDP2+FP8PM remains the better choice.

Recommended Configuration

For VERL GRPO on MI300X (ROCm) with FP8 memory savings:

strategy=fsdp2
rollout=sglang            # or vllm (with get_device_uuid fix)
FP8_PARAM_MANAGER=1
# Offloading now works with FP8PM (after clone fix):
#   param_offload=true  → -5% VRAM savings vs BF16 offload
#   param_offload=false → -25% VRAM savings vs BF16 (best)

For vLLM rollout, additionally set:

rollout.tensor_model_parallel_size=1   # TP≥2 may hang on ROCm
rollout.free_cache_engine=false
rollout.enforce_eager=true

ROCm Fixes

vLLM ROCm Fix

Missing @with_amdsmi_context decorator in vllm/platforms/rocm.py causes weight-transfer deadlock:

@classmethod
@with_amdsmi_context        # <-- ADD THIS LINE
def get_device_uuid(cls, device_id: int = 0) -> str:
    ...

FP8PM FSDP2 Offload Fix

save_for_backward pinned FSDP2's allgathered buffer (+44% regression). Fix in lumen/quantize/fp8_params.py:

# Change:
ctx.save_for_backward(fp8_weight, scale)
# To:
ctx.save_for_backward(fp8_weight.clone(), scale)

Megatron FP8PM Patch

FP8ParamManager for Megatron requires a patch to verl/workers/engine/megatron/transformer_impl.py to inject LumenConfig.enable() into _build_megatron_module(), after weight loading but before the module is returned. Gated by FP8_PARAM_MANAGER=1, applied only to the actor model (not the reference model). Megatron params stay in BF16 (on-the-fly quantization) to preserve distributed optimizer and DDP compatibility.
